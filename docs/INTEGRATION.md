# Building PulseKit & Using It in Another Project

How to build the PulseKit library and consume it from a **separate** app project.
Three methods, from most to least recommended.

---

## 0. What a consumer depends on

Two artifacts cover normal use (they pull the rest transitively):

| Artifact | Gives you |
|---|---|
| `io.pulsekit:pulse-android:0.1.0` | `PulseAndroid` init, the debug dashboard + notification launcher, FPS/Memory/Startup/Crash capture. Pulls `pulse-runtime → pulse-core → pulse-plugin`. |
| `io.pulsekit:pulse-network:0.1.0` | `PulseOkHttpInterceptor` for the API Requests panel. Only if you use OkHttp. |

Module dependency graph: `pulse-android`, `pulse-network` → `pulse-runtime` → `pulse-core`, `pulse-plugin`.

### Consumer build requirements (important)
Because `pulse-android` transitively uses **AndroidX Navigation 3**, the consuming app must use:
- **`compileSdk = 36`**
- **Android Gradle Plugin ≥ 8.9.1**
- **`minSdk ≥ 24`**
- Repositories: **`google()`** and **`mavenCentral()`** (Navigation 3 + AndroidX live on Google's Maven).

---

## Method A — Maven Local (recommended)

Publish the artifacts to your machine's `~/.m2` and depend on them by coordinate.

### 1. Publish from the PulseKit repo
```bash
cd /path/to/pulsekit
./gradlew publishToMavenLocal
```
This publishes `io.pulsekit:*:0.1.0` (verify: `ls ~/.m2/repository/io/pulsekit`).

### 2. In the consumer project
`settings.gradle.kts` — make sure `mavenLocal()` is a repository:
```kotlin
dependencyResolutionManagement {
    repositories {
        mavenLocal()   // ← PulseKit artifacts
        google()
        mavenCentral()
    }
}
```
App `build.gradle.kts`:
```kotlin
android {
    compileSdk = 36
    defaultConfig { minSdk = 24 }
}
dependencies {
    implementation("io.pulsekit:pulse-android:0.1.0")
    implementation("io.pulsekit:pulse-network:0.1.0") // optional (OkHttp capture)
}
```
Re-publish (`publishToMavenLocal`) whenever you change PulseKit.

---

## Method B — Gradle composite build (no publishing)

Best when you're editing PulseKit and the app **side by side** — changes are picked
up on every build with no publish step.

Consumer `settings.gradle.kts`:
```kotlin
includeBuild("/path/to/pulsekit")
```
Consumer `build.gradle.kts` — depend on it by the **same coordinates**; Gradle
substitutes them with the local build automatically:
```kotlin
dependencies {
    implementation("io.pulsekit:pulse-android:0.1.0")
    implementation("io.pulsekit:pulse-network:0.1.0")
}
```
(The modules declare `group = "io.pulsekit"` / `version = "0.1.0"` in the root
`build.gradle.kts`, which is what makes the substitution match.)

---

## Method C — Prebuilt AAR (least recommended)

Only if you can't use Gradle resolution. An AAR does **not** carry transitive
dependencies, so you must add every one by hand — error-prone. Prefer A or B.

```bash
./gradlew :pulse-android:assembleRelease :pulse-network:assembleRelease
# → pulse-android/build/outputs/aar/pulse-android-release.aar
# → pulse-network/build/outputs/aar/pulse-network-release.aar
```
Copy the AARs into the consumer's `libs/`, add a `flatDir` repo, then manually add
all transitive deps (compose, navigation3, coroutines, serialization, okhttp,
napier, androidx.core/lifecycle/activity…). Read them off the published `.pom`
files (`~/.m2/repository/io/pulsekit/**/*.pom`).

## Integrating in the app

```kotlin
// Application.onCreate()
class MyApp : Application() {
    override fun onCreate() {
        super.onCreate()
        PulseAndroid.initialize(this)   // debug-only by default
    }
}
```

```kotlin
// Capture HTTP for the API Requests panel (needs pulse-network)
val client = OkHttpClient.Builder()
    .addInterceptor(PulseOkHttpInterceptor())
    .build()
```

- **Opening the dashboard:** in a debug build PulseKit posts an ongoing notification;
  tapping it opens the dashboard. On **Android 13+** you must hold
  `POST_NOTIFICATIONS` (the permission is merged from the library; request it at
  runtime and then call `PulseAndroid.showDashboardNotification(this)` — see the
  sample's `MainActivity`).
- **Handled exceptions:** `PulseAndroid.recordException(throwable)`.
- **Custom events:** `Pulse.track("name", mapOf("k" to "v"))`.
- **Config:** `PulseAndroid.initialize(this) { enableMemory = true; notificationEnabled = true; … }`.
- **Debug-only:** with the default `debugOnly = true`, the notification/dashboard
  only appear in a debuggable build. Nothing user-facing ships in release.
- **Commit History panel:** the git-provenance asset is generated by the *app's*
  Gradle build. Copy the `generatePulseProvenance` task from `sample/build.gradle.kts`
  into your app module to populate that panel (optional).

## Publishing to a shared/remote repository (future)

For distribution beyond one machine, publish to Maven Central (group `io.pulsekit`)
per `ARCHITECTURE.md` §12. That needs Sonatype credentials, signing, and the
`maven-publish` config already present on each module extended with a remote
`repositories { }` block and the Vanniktech/Gradle publishing conventions. Until
then, Maven Local (Method A) or a composite build (Method B) is the way in.
